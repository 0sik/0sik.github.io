# WebSocket은 왜 필요할까

## 실시간 통신이 필요한 이유와 WebSocket의 동작 방식

웹 서비스를 만들다 보면 한 번쯤 이런 요구사항을 만나게 됩니다.

채팅 메시지가 오면 화면에 바로 보여야 하고,

주가나 코인 시세처럼 값이 계속 바뀌는 데이터는 새로고침 없이 갱신되어야 하며,

알림이나 모니터링 대시보드도 사용자가 보고 있는 동안 실시간으로 상태가 반영되어야 합니다.

이런 요구사항은 결국 하나의 질문으로 이어집니다.

**“웹은 원래 요청을 보내야 응답을 받는 구조인데, 서버가 먼저 클라이언트에게 데이터를 보내려면 어떻게 해야 할까?”**

이 지점에서 등장하는 기술이 바로 **WebSocket**입니다.

이번 글에서는 WebSocket이 무엇인지, 왜 필요한지, 그리고 HTTP와 무엇이 다른지를 정리해보겠습니다.

# SyncMon에 WebSocket이 필요했던 이유

## 실시간 모니터링에서 “바로 알려주는 것”의 중요성

WebSocket은 서버와 클라이언트가 연결을 한 번 맺은 뒤, 그 연결을 유지하면서 실시간으로 데이터를 주고받을 수 있게 해주는 기술입니다.

보통 웹은 HTTP 기반으로 동작합니다.

즉, 사용자가 요청을 보내야 서버가 응답합니다.

이 구조는 조회나 저장 같은 일반적인 기능에는 잘 맞지만, **서버에서 먼저 알려줘야 하는 상황**에는 불편합니다.

예를 들어 모니터링 시스템에서는 사용자가 새로고침을 하지 않아도, 이상 상황이 발생하는 순간 바로 화면에 반영되어야 합니다.

이때 매초 서버에 요청을 보내서 상태를 확인하는 방식으로도 구현은 가능하지만, 이벤트가 없을 때도 계속 요청이 발생하고, 이상 상황이 생겨도 다음 요청 시점까지는 지연이 생깁니다.

그래서 이런 경우에는 HTTP보다 **WebSocket처럼 연결을 유지하면서 서버가 즉시 push할 수 있는 구조**가 더 잘 맞습니다.

## SyncMon에서는 왜 WebSocket이 필요했을까

저희 SyncMon은 OGG 모니터링 시스템입니다.

이 시스템에서는 단순히 화면에 상태를 보여주는 것만으로는 부족한 시점이 있었습니다.

특정 모니터링 대상에서 이상이 발생하면,

운영자가 화면을 계속 보고 있지 않더라도 **즉시 인지할 수 있어야 했기 때문**입니다.

특히 요구사항 중 하나가

**서버에서 특정 모니터링 이상을 감지했을 때, 화면에서 TTS 음성이 바로 재생되어야 한다**는 것이었습니다.

이 요구사항을 HTTP 기반으로 풀려고 하면 구조가 애매해집니다.

클라이언트가 주기적으로 서버에

“이상 상황 생겼나요?”

를 계속 물어봐야 합니다.

그런데 이 방식은 두 가지가 불편합니다.

하나는 불필요한 요청이 계속 생긴다는 점이고,

다른 하나는 이상이 발생해도 polling 주기만큼 늦게 반응할 수 있다는 점입니다.

모니터링 시스템에서 이 지연은 생각보다 치명적입니다.

이상 상황은 “나중에 보여주는 것”보다 **“발생하자마자 알려주는 것”** 이 더 중요하기 때문입니다.

그래서 SyncMon에서는

서버와 화면 사이에 WebSocket 연결을 유지하고,

이상 이벤트가 발생하면 서버가 해당 이벤트를 즉시 클라이언트로 push하도록 구성했습니다.

클라이언트는 이 이벤트를 받으면

화면 경고를 띄우고, 필요한 경우 TTS를 통해 음성으로도 알리도록 처리할 수 있습니다.

즉, 여기서 WebSocket을 쓴 이유는

단순히 “실시간 기술을 써보고 싶어서”가 아니라,

**모니터링 이상을 운영자가 놓치지 않도록 하기 위해서**였습니다.

---

## 정리

WebSocket은 모든 웹 기능에 필요한 기술은 아닙니다.

하지만 **서버에서 발생한 이벤트를 즉시 사용자 화면에 전달해야 하는 경우**에는 굉장히 유용합니다.

SyncMon에서도 WebSocket이 필요했던 이유는 명확했습니다.

- 이상 상황은 서버가 먼저 알고
- 그 정보를 화면에 즉시 전달해야 하며
- 운영자는 그 순간 바로 인지해야 했기 때문입니다

특히 TTS처럼 **이벤트가 발생하자마자 즉시 반응해야 하는 기능**은

요청-응답 기반보다 서버 push 방식이 훨씬 자연스럽습니다.

결국 SyncMon에서 WebSocket은

실시간 통신 자체가 목적이 아니라,

**장애 징후를 더 빠르고 명확하게 전달하기 위한 수단**이었습니다.
